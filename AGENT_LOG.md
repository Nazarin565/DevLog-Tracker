### Manual interventions and why

1. **`AgentResult` type defined locally in `lib/api.ts`** ‚Äî the type lives in `packages/server/src/agents/types.ts` and is not exported from `@devlog/shared`. Rather than leaking a server-internal type into shared, defined a local interface `AgentResult<T>` with `output` and `steps` in the web package. This keeps the server/web boundary clean.

2. **`providers.tsx` created after initial `layout.tsx`** ‚Äî the agent initially referenced `./providers` in `layout.tsx` before creating the file. IDE flagged `TS2307`. The providers file was created immediately after; this is a normal file-ordering artefact of sequential generation.

3. **`packages/server/.env` and `packages/web/.env.local` created for local dev** ‚Äî these are gitignored. Without them the server defaults are correct (`LLM_PROVIDER=mock`) but the port/CORS config needed explicit values to start cleanly. These files are not committed; `.env.example` at root documents all variables.

4. **`useRunAgent` signature refactored ‚Äî `agentId` moved to call-site** ‚Äî the agent-generated version accepted `agentId` as a constructor argument (`useRunAgent<T>(agentId)`), producing a hook instance tied to one agent. This meant `AgentPanel` needed a separate hook instance per agent, which is awkward with conditional rendering. Fixed manually: `useRunAgent<T>()` takes no argument; `agentId` is passed together with `input` at `mutateAsync({ agentId, input })` time. Result: one hook instance, any agent can be called from it.

5. **Null-guards added to `AgentOutput` in `AgentPanel`** ‚Äî the agent-generated code cast `output` to the expected type without validation. If the API ever returns an unexpected shape, the component would crash with a runtime type error. Added explicit guards (`if (!result || typeof result.type !== 'string')` and `if (!pResult || !Array.isArray(pResult.rankedTasks))`) that render a fallback error message instead of throwing.

6. **`AgentOutput` null-guards replaced with Zod `safeParse`** ‚Äî the manual guards from fix 5 were replaced with `DecompositionResultSchema.safeParse(output)` and `PrioritisationOutputSchema.safeParse(output)`. This is strictly better: the same schemas used on the server now validate the output on the client, consistent with the project invariant "never trust LLM output, always safeParse". Manual `typeof` checks would silently pass structurally wrong data that Zod would catch.

7. **`taskTitleMap` moved from internal `useTasks()` call to prop** ‚Äî the agent-generated `AgentPanel` fetched all tasks itself via `useTasks()` to resolve task IDs to titles. Fixed: `taskTitleMap` is now passed as a prop from `[id]/page.tsx`, which already has `useTasks()` in scope. This avoids a redundant duplicate network request inside the component and keeps data fetching at the page level.

## Step 10 (post-scaffold) ‚Äî Code Review Fixes

**Date:** 2026-05-21

### Context

After scaffolding and integration (steps 9‚Äì10), a structured senior fullstack review was performed across all layers: shared contracts, db, routes, agent framework, llm layer, and web. The review used a checklist covering API design, validation, DB edge cases, type correctness, and frontend state/rendering. Six issues were found and fixed in this session. All fixes were made manually ‚Äî the agent (Claude Code) identified the issues; changes were human-confirmed before writing.

### Fixes applied

1. **`order` param ignored for `sortBy=priority`** ‚Äî `taskRepository.ts:72`. The `dir` variable was computed from `query.order` but hardcoded to `DESC` in the priority branch. Fixed: `ORDER BY ${PRIORITY_ORDER} ${dir}, created_at DESC`. Users selecting `sortBy=priority&order=asc` now get low‚Üímedium‚Üíhigh.

2. **Agent route passed `req.body` to agents without validation** ‚Äî `routes/agents.ts:22`. Raw `req.body` was forwarded directly to agents, so a malformed body (e.g. missing `taskId`) would cause an opaque runtime error inside the agent instead of a clean 400. Fixed: added `DecomposeInputSchema` and `PrioritiseInputSchema` to `packages/shared/src/agents.ts`; `routes/agents.ts` now looks up the schema by `agentId` and calls `.parse(req.body)` before passing input to the agent. ZodError ‚Üí 400 is handled by the existing `errorHandler` middleware. This enforces the project invariant "validate at system boundaries."

3. **`api.subtasks.create` typed as `Promise<Task>` instead of `Promise<Subtask[]>`** ‚Äî `web/lib/api.ts:58`. The server route `POST /api/tasks/:id/subtasks` returns `Subtask[]` (201), but the client wrapper was typed as `Promise<Task>`. TypeScript did not catch this because `request<T>` uses `as Promise<T>`. Fixed: changed to `Promise<Subtask[]>` in `api.ts` and added explicit generic `useMutation<Subtask[], Error, CreateSubtasksInput>` in `useTasks.ts`.

4. **`SubtaskProposal.description` silently dropped on subtask creation** ‚Äî `shared/src/agents.ts:17`. `SubtaskProposalSchema` had an optional `description` field, but `CreateSubtasksSchema` and the DB schema only accept `title`. Any `description` returned by the LLM was silently discarded. Fixed: removed `description` from `SubtaskProposalSchema` ‚Äî the LLM prompt does not ask for it, the DB does not store it, so the field was phantom. Keeping a field that is always dropped is misleading; the cleaner fix is to remove it rather than expand DB schema for unused data.

5. **Empty `IN()` edge case in `taskRepository.findAll`** ‚Äî `taskRepository.ts:80`. When `findAll` returned zero task rows, the subtask query generated `WHERE task_id IN (NULL)` via a `|| 'NULL'` fallback. This worked in SQLite but was an opaque workaround. Fixed: added an early `if (rows.length === 0) return []` guard before the subtask query, and removed the `|| 'NULL'` fallback. The IN-clause now only runs when there are actual task IDs.

6. **`key={i}` (array index) in AgentPanel lists** ‚Äî `AgentPanel.tsx:100, 140, 153`. Three lists used index as React key: reasoning steps, clarify questions, proposed subtasks. Fixed: changed to composite keys (`${step.label}-${i}`, `q-${i}`, `st-${i}`) which are stable within a single agent result.

### What was NOT fixed (intentional scope decisions)

- **No routes for `subtaskRepo.setDone`/`remove`**: `subtaskRepository.ts` has `setDone` and `remove` methods but no corresponding HTTP routes. `SubtaskList` displays done/pending state but cannot toggle it from the UI. Kept as a conscious scope decision ‚Äî adding toggle would require a new route (`PATCH /api/tasks/:id/subtasks/:subId`) and UI affordance. Flagged as a future improvement, not a bug.
- **Non-deterministic subtask ordering**: `createMany` sets a single `now` timestamp for all subtasks in a batch, so `ORDER BY created_at ASC` between them is non-deterministic. Left as-is ‚Äî the practical impact is negligible and fixing it would require either a sequence column or inserting timestamps with millisecond offsets.
- **`confirm()` for delete**: two pages use native `window.confirm`. Functional, but not styled and can be suppressed in some environments. Left as-is for scope.

### Result

- `npm run typecheck` ‚Üí exit 0 (all packages)
- `npm test` ‚Üí 14/14 tests pass

## Step 11 ‚Äî Antipatterns, Bottlenecks & Production Risk Audit

**Date:** 2026-05-21

### Context

Full static analysis + golden path runtime verification (Playwright headless Chromium + API smoke tests). Goal: surface anything that could break in production or under realistic load before the project is submitted.

### Static analysis findings

**Not critical for this project's scope, but documented:**

1. **No LLM timeout** ‚Äî `AnthropicClient.complete()` makes a bare `client.messages.create()` call with no `AbortSignal` or server-side timeout. If the Anthropic API hangs, the Express request hangs indefinitely. For a local single-user tool this is acceptable; in production it would need a `signal: AbortSignal.timeout(30_000)` wrapper.

2. **Agent errors surface as opaque 500** ‚Äî when the LLM returns non-JSON or fails Zod validation, the agent throws `new Error(...)`. The `errorHandler` catches it and returns `{ error: { message: 'Internal server error', code: 'internal_error' } }` ‚Äî the real error message is `console.error`'d on the server but hidden from the client. This is correct security behaviour (no LLM output leaked to client), but the UI shows the same "Internal server error" for a network failure, a missing taskId, and a Zod parse failure ‚Äî no distinction. Acceptable for scope.

3. **No graceful shutdown** ‚Äî `server.listen(port, ...)` with no `SIGTERM`/`SIGINT` handler. SQLite `better-sqlite3` is synchronous so no in-flight writes are lost on kill, but any pending LLM request would be dropped without a clean error. Low risk for a local tool.

4. **express.json() with no body size limit** ‚Äî default Express v5 limit is 100 kb. Realistically not a problem for this app's payloads (task descriptions, subtask lists).

5. **No indexes on `status`, `priority`, `created_at`** in SQLite schema. With tens of tasks this is irrelevant; with thousands it would matter. Noted as future-scale concern only.

6. **SQL injection surface is safe** ‚Äî dynamic `UPDATE tasks SET ${fields.join(', ')}` looks dangerous but is safe: `fields` contains only hardcoded strings (`'title = ?'`, `'status = ?'` etc.), never user input. `sortBy`/`order` query params are constrained by `z.enum` in `TaskQuerySchema` before they reach the repository. No injection vector found.

7. **CORS default is safe** ‚Äî `WEB_ORIGIN` defaults to `http://localhost:3000`, not `*`. No wildcard CORS.

8. **No secrets in web** ‚Äî only `NEXT_PUBLIC_API_URL` is exposed to the browser. `ANTHROPIC_API_KEY` is server-only.

## Step 12 ‚Äî Manual UX fixes after runtime audit

**Date:** 2026-05-22

Both changes below were identified during the Step 11 runtime audit (Playwright golden path) and fixed manually. No agent generation involved.

### Fix A ‚Äî Interactive subtask checkboxes with persistent state

**Problem:** `SubtaskList` rendered checkbox-shaped spans that were purely decorative ‚Äî no `onClick`, no route behind them. The `done` field existed in the DB and the repository had `setDone`, but there was no HTTP route to reach it. Identified as a conscious scope decision in Step 10, but flagged as a UX issue during runtime: a reviewer clicking the checkboxes would expect them to work.

**What was added:**

- `PATCH /api/tasks/:taskId/subtasks/:subId` route in `packages/server/src/routes/tasks.ts` ‚Äî validates `done: boolean` in the body, delegates to `subtaskRepo.setDone`, returns the updated `Subtask`. Returns 404 if task or subtask not found.
- `api.subtasks.setDone(taskId, subId, done)` in `packages/web/lib/api.ts`.
- `useToggleSubtask(taskId)` mutation hook in `packages/web/hooks/useTasks.ts` ‚Äî on success invalidates `['task', taskId]` and `['tasks']` so both the detail page and the task list card update.
- `SubtaskList` rewritten: each row is now a `<label>` wrapping a visually hidden `<input type="checkbox">` and the decorative span. Clicking the label or the span toggles `done`. The `disabled` prop on the input prevents double-toggle while the mutation is in flight. `taskId` added as a required prop; all two call-sites (`TaskCard`, `[id]/page.tsx`) updated.

### Fix B ‚Äî `cursor: pointer` on filter/sort/order pill buttons

**Problem:** The status / sort / order pill buttons in `app/page.tsx` had no `cursor-pointer` class. In Tailwind v4 `<button>` elements do not get `cursor: pointer` automatically (unlike some browser defaults), so hovering the filter bar showed the default text cursor ‚Äî no visual affordance that the pills are clickable.

**What was changed:** Added `cursor-pointer` to the `className` of all three button groups (status, sortBy, order) in `packages/web/app/page.tsx`.

### Result

- `npm run typecheck` ‚Üí exit 0
- `npm test` ‚Üí 14/14 (no server-side test changes needed; toggle route logic is trivial)

## Step 14 ‚Äî AI features redesign: Decompose in SubtaskList + Prioritise on home page

**Date:** 2026-05-22

All changes below were made manually. No agent generation involved.

### Context

Two AI features existed previously but were architecturally misplaced: Decompose and Prioritise both lived inside `AgentPanel` on the task detail page. This step relocates them to where they belong UX-wise and makes both interactions self-contained.

### Decompose ‚Äî moved into SubtaskList

**Problem:** Decompose was a tab inside `AgentPanel`, a separate purple block below the subtask list. It was visually disconnected from the subtasks it produces.

**What was changed:**

- `components/SubtaskList.tsx` ‚Äî added full decompose flow directly inside the subtask list:
  - A full-width `‚ú¶ Decompose task` button (purple, `rounded-lg`) sits below `+ Add subtask` as a separate row.
  - Clicking runs the decompose agent (`useRunAgent`, `agentId: 'decompose'`). Button text changes to "Decomposing‚Ä¶" (disabled) while pending.
  - If existing subtasks are present, the button runs the agent immediately ‚Äî no blocking confirm dialog. Instead, after the agent returns proposed subtasks, a small warning line appears between the list and Save: `‚öÝ Saving will replace your N existing subtask(s).`
  - If the agent returns `type: 'clarify'` ‚Äî shows an amber block with clarifying questions and a Dismiss button.
  - If the agent returns `type: 'subtasks'` ‚Äî shows a purple block "Proposed subtasks" with the list, the warning (if applicable), and two buttons: "Save subtasks" / "Discard".
  - **Save subtasks** ‚Äî first deletes all existing subtasks (`api.subtasks.remove` in parallel), then calls `createSubtasks`. Old subtasks are only deleted at save time, not before. If the user clicks Discard or navigates away, old subtasks are untouched.
  - Zod `DecompositionResultSchema.safeParse` used on the agent output ‚Äî consistent with the project invariant.
- `components/AgentPanel.tsx` ‚Äî **deleted**. No longer needed.
- `app/tasks/[id]/page.tsx` ‚Äî removed `AgentPanel` import and render, removed `useTasks` / `allTasks` that were only used to build `taskTitleMap` for the panel.

### Prioritise ‚Äî moved to home page with localStorage persistence

**Problem:** Prioritise was also inside `AgentPanel` on the detail page ‚Äî wrong place for a feature that ranks all tasks globally.

**What was changed:**

- `hooks/usePrioritisation.ts` ‚Äî new hook file:
  - `usePrioritisationResult()` ‚Äî reads `StoredPrioritisation` (`{ rankedIds, summary, savedAt }`) from `localStorage` key `devlog:prioritisation` via a TanStack Query query (`queryKey: ['prioritisation-result']`, `staleTime: Infinity`). Survives page reload.
  - `useRunPrioritisation()` ‚Äî runs the prioritise agent, validates output with `PrioritisationOutputSchema.safeParse`, stores result to localStorage and updates QC cache via `setQueryData`. Throws on parse failure.
  - `useClearPrioritisation()` ‚Äî removes localStorage entry and sets QC cache to `null`.
- `components/PrioritisePanel.tsx` ‚Äî rewritten as `PrioritiseButton`: a single compact `‚ú¶ AI Prioritise` button in the page header. Shows "Prioritising‚Ä¶" when pending.
- `app/page.tsx` ‚Äî full redesign of the home page AI integration:
  - `PrioritiseButton` sits in the top-right header row alongside `+ New Task`.
  - After a successful run, a `‚ú¶ AI` pill appears in the Sort filter row (purple, alongside Priority / Date pills).
  - Clicking `‚ú¶ AI` sets `sortBy=ai` in the URL. The task list is then sorted client-side: tasks are reordered by `rankedIds` index from the stored result; tasks not in the ranked list fall to the end.
  - While `sortBy=ai` is active, the Order (asc/desc) control is hidden (AI order is fixed).
  - Below the filter row, a status line shows: "Sorted by AI ¬∑ {date} ‚Äî {summary}" with an ‚úï button that clears the result from localStorage + QC cache and resets `sortBy` to `createdAt`.
  - The `‚ú¶ AI` pill only appears if a stored result exists ‚Äî first run required.

### Architectural decision: localStorage vs server

Prioritisation result stored client-side in localStorage (Variant C), not in the database (Variant D). Rationale: this is a single-user local tool; localStorage survives reloads and requires no schema changes. Variant D (server persistence) was discussed and noted as the production-appropriate choice for a multi-user deployment.

- `npm run typecheck` ‚Üí exit 0
- `npm test` ‚Üí 14/14
